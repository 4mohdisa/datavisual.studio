# datavisual.studio — Ship It (definitive end-to-end plan)

**One session to done.** End-to-end test every feature, make the monorepo deploy-ready, fix whatever's
broken, and leave it so the only remaining human step is pasting Clerk production keys into Vercel.

This is interactive where it must be (account sign-ups, the AWS box, DNS) and autonomous everywhere
else. Read the two settled decisions first — they've caused repeated churn and are now final.

---

## Two decisions that are FINAL — do not reopen

**1. Storage is JSON on disk. There is no Postgres and no Supabase, tonight or as a follow-up.** The
architecture is deliberate: JSON files under `data/`, behind one storage module. This is a portfolio
project and an early-users product; JSON on a persisted disk runs both correctly. A database earns its
place only when multiple backend replicas run at once — which is not this project's situation. "No
database, by design" is a documented, defensible engineering choice. `sqlalchemy`/`psycopg2`/`pymysql`
in the dependencies are **only** for the user-facing data connectors (a user importing *their own* DB as
a data source) — they are not this app's storage. **Do not migrate anything.**

**2. The assistant's "leash" stays. It is not a spending limit and BYO keys do not change it.** The
discipline is: *numbers are computed by pandas, never generated by the model.* The model writes a query
spec, the engine executes it, the model phrases the real result. This is what prevents the model
free-handing arithmetic — the exact failure that once reported "weekly earnings = $480,506." It is
correct, it is the same pattern every serious data tool uses, and it is the project's best engineering
story. **General/meta questions ("what is this about?") are already answered in plain language and were
never leashed** — the leash only ever applied to numeric claims. So there is nothing to "unleash"; the
task is to confirm the assistant works well across every kind of question, not to loosen it.

## The likely reason the chatbot "still isn't fixed"

The meta-question fix shipped two sessions ago and is browser-verified: "what is this about?" returns a
real summary. **But a stale Next dev server is running on port 3000, while the app runs on 3100.**
Anyone testing on `localhost:3000` has been hitting an old build. **Kill port 3000 at the start of this
session** so nobody — human or agent — tests the wrong thing again.

## Hard rules

JSON storage. No new runtime dependencies. `main` only, no force pushes. Failing test first for any fix.
A green tick needs re-checkable evidence — a screenshot, a saved artifact, a command someone can re-run.
Never run `next build` while the dev server is live on the same `.next`.

# PHASE 1 — End-to-end feature test (find everything broken before deploying)

Drive the **real app in a real browser** and produce **`E2E_REPORT.md`** — one row per feature: works /
loading state / error state / empty state / mobile 390 / screenshot. Fix everything red, failing-test
first. This is the "make sure everything actually works" the owner keeps asking for.

**Data & dashboards:** CSV upload (small, and >5MB via the direct-upload path); dashboard auto-build;
every metric card; **all 9 chart types individually rendered and eyeballed** (line, bar, scatter,
histogram, pie, box, heatmap, area, treemap); the data table (search, sort, paginate, CSV export);
entity comparison (2-metric and 3+-metric).

**The assistant — exhaustively, because this is the owner's #1 pain:**
- Data questions: "total MRR", "which plan grew fastest", "average revenue per customer", "how many
  customers in June" (must total across plans, not silently answer one). Numbers correct, working shown.
- Meta questions: "what is this about?", "what can you do?", "explain this dashboard", "what data is
  this?". Plain-language answers, no error.
- Edits: "add a bar chart of revenue by plan", "remove the pie chart". Status reads "Updating the
  dashboard…", not a lie.
- The first-message set a real stranger sends: "hello", "help", empty send, a single "?", a nonexistent
  column, a nonsense string. **Every one returns something useful — none may error or dead-end.**
- Status always follows intent; no path shows "Updating dashboard" while answering; no error says "edit
  failed" for a non-edit. Pin-as-metric works.

**Research pipeline:** a full run to completion; report renders; sources cited; follow-ups work.

**Sharing & export:** mint a share link, open it in a clean incognito window, scroll it, revoke it,
confirm it 404s; PDF export (light, complete, not blank — watch backend memory during it) and HTML
export.

**Connectors:** a real SQL DB and a REST endpoint end-to-end. Confirm the SSRF guard blocks
metadata/localhost but a *legitimate* connector still works and says so clearly when refused.

**Everything else:** `/demo` (anonymous, scrolls); onboarding for a brand-new user (lands somewhere
useful, not an empty room); the account/API-key panel (add, validate, masked on reload); `/admin`
(renders, events flowing); the error paths — backend down, no key, bad CSV, rate-limited — each a clear
message, never a white screen.

# PHASE 3 — Monorepo deploy-ready (so tomorrow is paste-keys-and-go)

The two-origin production shape (Vercel frontend ↔ AWS backend) has never actually booted. Prove it, and
fix what breaks — two real deploy bugs were already caught this way (`docker-compose.yml` missing
`SECRET_KEY`; the frontend Dockerfile ignoring the build-time-inlined `NEXT_PUBLIC_BACKEND_ORIGIN`).

- `docker compose` builds **both** with production env: `PROXY_SHARED_SECRET`, `SECRET_KEY`,
  `ALLOWED_ORIGINS` locked to the app origin (no wildcard), `TRUSTED_PROXY_HOPS`, and
  `NEXT_PUBLIC_BACKEND_ORIGIN` pointing at the backend as a **genuinely separate origin**.
- Exercise what only breaks across origins: CORS preflight on the direct-upload path (allowed from the
  app origin, refused from an unknown one), the HMAC upload ticket with a >5MB file, the polling
  pipeline to completion, `/health` through the proxy.
- **`make verify-deploy` against the two-origin stack** — not the single-origin dev one.
- `data/` survives a container restart; `next.config` standalone stays gated on `DOCKER_BUILD=1` so
  Vercel uses its native build.
- **CI green on GitHub** — check `github.com/4mohdisa/datavisual.studio/actions` (no `gh` needed to
  *look*). If red, read the failing step and fix; the suite is now hermetic so the likely causes are
  Playwright browser deps or the e2e port, not a model dependency. A green run URL goes in `HANDOFF.md`.

# PHASE 4 — The deploy runbook a human follows tomorrow

`DEPLOY_RUNBOOK.md`, numbered and copy-pasteable, so the owner's launch is mechanical. It must state,
in order:

1. **Clerk production instance** — create it, add the domain, set redirect URLs, copy `pk_live_`/
   `sk_live_`. (Start this early — it's the only step that can wait on DNS.)
2. **Backend on the AWS box** — the `free -h`/`df -h` headroom check first (idle ~85 MB, PDF export peaks
   ~480 MB with Chromium; under ~600 MB free, add ≥1 GB swap); `docker compose` up; port 8001;
   persisted `data/` with `SECRET_KEY` **alongside it**; single replica; `curl /health`.
3. **HTTPS** — an `api.` subdomain fronted by the existing reverse proxy; don't expose 8001 publicly;
   `TRUSTED_PROXY_HOPS` matching the real chain; IMDSv2 enforced.
4. **Vercel** — root `frontend`; **all `NEXT_PUBLIC_*` and the `pk_live_`/`sk_live_` keys and
   `PROXY_SHARED_SECRET` (identical to the backend) set BEFORE the first build**; deploy; attach domain.
5. **Verify live** — `make verify-deploy` against the real host; the A/B walk on production; the
   first-message set on the live `/demo`.
6. **Before announcing** — install the `data/` backup cron and `make gc`. `data/` is the only copy;
   nothing else holds a backup.
7. A short **troubleshooting table** — everything-403s → `PROXY_SHARED_SECRET` mismatch; silent sign-in
   fail → test keys or domain not added; upload fails in prod → `ALLOWED_ORIGINS`; backend won't boot →
   `SECRET_KEY` missing/mismatched; everyone rate-limited → `TRUSTED_PROXY_HOPS`; backend dies on export
   → memory.

**Say plainly in the runbook that no database is installed, and why the SQL libraries are nonetheless
present** — so nobody provisions a database that isn't needed.
